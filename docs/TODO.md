- [x] **Early-assignment surfacing + chain-builder correctness pass (2026-06-27)** — triggered by "I got assigned 100 NVDA early but the dashboard never told me." Turned into a multi-bug investigation reconciled against the user's Excel (`OneDrive/Documents/Investments/Options tracker - Guru.xlsx`, the source of truth; reconcile as **Excel points × 100 = app commission-net dollars**). All changes are in **`backend/app/analytics/rolls.py`** (chain builder) and **`backend/app/poller/jobs/rolls.py`** (rebuild job) unless noted. **96 backend tests pass.** Backend **and** frontend images were rebuilt+restarted at the end (`docker compose up -d --build backend frontend`).

  **What was wrong & what was fixed (in order discovered):**
  1. **Live-poll assignments were dropped.** IBKR's live `/iserver/account/trades` feed reports an assignment as `side="A"` (e.g. NVDA `0001bcd1.6a3ccf2c.03.01`, null strike, `order_description:"ASSIGNED 1 on OCC"`), but `build_roll_chains` only branched on `S`/`B`. Fix: new module helper `_is_assignment(e)` (true for `side=="A"` **or** Flex `raw.notes/code=="A"`); the option close branch is now `elif side in ("B","A")`. When the assignment row's strike is null, the chain is recovered by `conid` (`_find_open_chain_by_conid`) then by underlying+right (`_find_open_chain_by_underlying_right`). For a put assignment with no real STK row, a synthetic `assignment_stock` leg is created (shares = contracts×100 @ chain strike) — guarded by `stk_assignment_underlyings` (underlyings that DO have a real STK assignment exec, e.g. from Flex) so shares aren't double-counted. `side=="A"` also handled in the STK branch.
  2. **Duplicate assignment → phantom "NVDA P" (strike-0) chain.** The same assignment arrives from BOTH Flex (`OPT side B`+`notes A`) and the poll (`OPT side A`); `dedupe_executions` keys on `(conid,side,qty,price)` so it can't match `A`↔`B` and both survive. The poll duplicate found no open short and fell into the orphan branch, minting a strike-0 chain. Fix: an assignment that matches no open chain is now a **no-op (`continue`)**, never an orphan.
  3. **`build_rolls` was crashing every cycle (root cause of "nothing updates").** The wipe `DELETE FROM roll_chains WHERE is_manual IS false` was blocked by a FK from `chain_adjustments` (no cascade), so the whole rebuild transaction rolled back every 20s — chains had been frozen at 2026-06-16 while executions ran to 2026-06-26. The blockers were 3 corrupt `manual_link` rows the user created 2026-06-22 (two duplicates + one pointing at a since-deleted exec, all linking the *closed* 216 chain into the 215 chain). **One-time data fix (with user approval):** `DELETE FROM chain_adjustments WHERE id IN (1,2,3)` — after which the running (old-image) job self-healed. **Durable code fix:** `build_rolls` now **upserts rebuilt chains first, then deletes only stale chains** (chain_id not in the rebuilt set), clearing any adjustments that point at stale chains first. So a future manual link can never freeze the pipeline. NOTE: `is_manual` is effectively always false (nothing sets it true; manual edits live in `chain_adjustments` and are re-applied each build via the deterministic chain_id).
  4. **OptionEAE expirations re-opened closed chains (Fix A).** See the corrected note in "polish / follow-ups": EAE Expiration `quantity` sign is `+1` puts / `-1` calls, so puts imported as `side="S"`. Now a `flex_eae` row is intercepted **before** the S/B/A dispatch and only ever *closes* the open leg (`role="expired"`, moves `_opt_pos` toward 0); if the contract is already flat it's a no-op. A `$0` buy with nothing to close is also skipped (duplicate-expiry twin), and a `$0` buy-to-close is labelled `expired`. This fixed QQQM 225 & 235 (were wrongly OPEN) and removed two QQQM-210 $0 phantoms.
  5. **Cross-strike rolls were split into two chains (Fix B).** Chains key on `(ticker,right,strike)`, so `NVDA 216→215` became a closed 216 + open 215 instead of one rolled chain (as the sheet models it). Fix: in the `side=="S"` path, before minting a new chain, `_find_roll_candidate` looks for a same-`(underlying,right)` chain that went flat via a buy on the **same calendar day** at a different strike; if found it revives that chain as a roll — relabels its closing leg `close→roll` (`_relabel_last_close_as_roll`), appends the new sell as `role="roll"`, moves it in `active_chains` to the new strike key, and updates `strike`/`_current_expiry`/`_last_conid`. Constant `ROLL_WINDOW_DAYS=1`. The chain_id stays the original-open hash.
  6. **Latent signed-qty bug exposed by Fix B.** Flex stores **signed** quantity (sells `-1`); the position tracker did `_opt_pos -= qty`, so a sold put was tracked as *long* (`_opt_pos` went +). Dollars were always right (`_credit` uses `abs`), but the broken `_opt_pos` blocked roll detection. Fix: `abs(float(e.qty or 0))` in all three `_opt_pos` updates (open, buy/close, orphan). Also added a sort tiebreak so on tied (date-only Flex) timestamps **buys sort before sells**, so a roll's buy-to-close is seen before the new sell.

## Known Issues / Operations

### IBEAM authentication instability
**Status:** Resolved for now (2026-06-20), but fragile.
**Risk:** IBKR periodically updates their login page HTML, breaking IBEAM's hardcoded CSS selectors.
If login breaks:
1. Check `gateway/outputs/` for failure screenshots (mounted volume).
2. Inspect the login page HTML to find current element IDs/names.
3. Update the selectors in `docker-compose.yml` under `ibkr-gateway` environment.

### Flex Web Service — one-time setup required
The `import_flex_trades` job calls IBKR's Flex Web Service directly. For this to work, the user must:
1. Log into IBKR Client Portal → Settings → Flex Web Service → Enable → Generate token
2. Reports → Flex Queries → Create Activity Flex Query (Sections: Trades + Option Exercises, Assignments and Expirations, Format: XML, Period: Since Inception, Breakout by Day: No)
3. Copy the token to `IBKR_FLEX_TOKEN=` in `.env`
4. Copy the query ID to `IBKR_FLEX_QUERY_ID=` in `.env`

### CSV upload parser — IBKR format sensitivity
The `csv_import.py` parser handles the standard IBKR Activity Statement CSV format, but IBKR may change column headers or date formats. If the CSV import fails, check the column headers in the downloaded file against the parser's expected keys.

### `flex_parse.py` note
The file `backend/app/clients/ibkr/flex_parse.py` handles the XML format from IBKR's Flex Web Service and IS used by `flex_web.py`. **Do not delete** — it is actively imported by `flex_web.py:fetch_flex_trades()`.
